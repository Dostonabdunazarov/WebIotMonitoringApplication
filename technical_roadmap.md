# Technical Roadmap

## 1. Цель

`WebIotMonitoringApplication` должен стать backend-сервисом мониторинга и анализа IoT-устройств в реальном времени, входящим в состав общей системы.

Итоговая цель roadmap:

- принимать телеметрию;
- хранить и агрегировать данные в `PostgreSQL`;
- поддерживать realtime-мониторинг;
- генерировать алерты и события;
- предоставлять API для UI и внешних сервисов;
- быть готовым к интеграции в общую платформу.

## 2. Общая стратегия

Развитие проекта рекомендуется вести поэтапно:

1. Сначала подготовить устойчивую backend-основу.
2. Затем реализовать базовый контур мониторинга.
3. После этого добавить realtime и аналитику.
4. В финале усилить интеграции, наблюдаемость и эксплуатационную готовность.

## 3. Этап 0. Foundation

### Цель

Подготовить проект к роли backend-сервиса.

### Задачи

- перевести текущее приложение из шаблонного MVC в backend-first формат;
- определить структуру слоёв `Presentation`, `Application`, `Domain`, `Infrastructure`;
- привести конфигурацию `Program.cs` к API-ориентированной архитектуре;
- добавить `Swagger/OpenAPI`;
- подключить `Health Checks`;
- подготовить базовую конфигурацию окружений;
- зафиксировать соглашения по логированию и обработке ошибок.

### Результат

- проект готов к развитию как backend API;
- есть единая структура решения;
- есть базовый инфраструктурный каркас.

### Definition of Done

- приложение стартует как API;
- `Swagger` доступен в dev-среде;
- health endpoint работает;
- структура папок или проектов утверждена.

  - `MonitoringRule`

- настроить fluent-конфигурации для таблиц;
- добавить индексы для hot-path запросов;
- подготовить первые миграции;

- определить стратегию работы с `jsonb`;
- предусмотреть будущую партиционизацию таблиц телеметрии.

### Результат

- есть начальная схема БД;
- доменные сущности отражены в коде;
- база готова к приёму первых данных.

### Definition of Done

## 5. Этап 2. Device Registry

### Цель

Реализовать реестр устройств и их метаданные.

### Задачи

- реализовать CRUD для устройств;
- поддержать типы устройств и профили;
- хранить внешний идентификатор устройства;
- хранить статус активности и `last_seen_at`;
- добавить фильтрацию и пагинацию списка устройств;
- предусмотреть связь устройства с локацией или объектом;
- ввести базовую валидацию входных DTO.

### Результат

- система умеет регистрировать и обслуживать каталог устройств;
- UI и внешние сервисы могут получать список устройств.

### Definition of Done

- доступны endpoints:
  - `GET /api/devices`
  - `GET /api/devices/{id}`
  - `POST /api/devices`
  - `PUT /api/devices/{id}`
- есть тесты на базовые сценарии CRUD.

## 6. Этап 3. Ingest телеметрии

### Цель

Сделать надёжный приём телеметрии от устройств или gateway.

### Задачи

- реализовать `POST /api/ingest/telemetry`;
- реализовать `POST /api/ingest/batch`;
- валидировать источник и структуру payload;
- сохранять сырые сообщения в `telemetry_messages`;
- извлекать нормализованные показатели в `telemetry_readings`;
- поддержать идемпотентность повторной отправки;
- логировать невалидные или неподдерживаемые payload;
- обновлять `DeviceState` при успешном ingest.

### Результат

- backend начинает принимать реальные данные;
- появляется единый входной контур телеметрии.

### Definition of Done

- ingest endpoint принимает одиночные и batch-сообщения;
- данные попадают в БД корректно;
- состояние устройства обновляется после нового сообщения.

## 7. Этап 4. Device State и Online Monitoring

### Цель

Сделать быстрый доступ к текущему состоянию устройства.

### Задачи

- реализовать таблицу и модель `DeviceState`;
- обновлять online/offline статус;
- хранить последние значения ключевых метрик;
- отслеживать `last_seen_at`;
- добавить heartbeat endpoint при необходимости;
- реализовать фоновую проверку устройств, потерявших связь;
- подготовить быстрые read endpoints для UI.

### Результат

- можно быстро понять, что происходит с устройством прямо сейчас;
- операторский интерфейс получает актуальное состояние без тяжёлых запросов.

### Definition of Done

- доступны endpoints:
  - `GET /api/devices/{id}/state`
  - `POST /api/devices/{id}/heartbeat`
- offline-логика работает по таймауту;
- текущее состояние читается без сканирования всей телеметрии.

## 8. Этап 5. Rules, Events и Alerts

### Цель

Добавить активный мониторинг и реакцию на проблемы.

### Задачи

- реализовать сущность `MonitoringRule`;
- поддержать правила по порогам и пропаданию связи;
- создать контур генерации `DeviceEvent`;
- реализовать жизненный цикл `Alert`;
- добавить подтверждение и закрытие алертов;
- сохранять историю смены статусов;
- подготовить API для списка активных алертов.

### Результат

- система не только хранит телеметрию, но и реагирует на нарушения;
- появляется операционный контур инцидентов.

### Definition of Done

- доступны endpoints:
  - `GET /api/alerts`
  - `GET /api/devices/{id}/alerts`
  - `POST /api/alerts/{id}/acknowledge`
  - `POST /api/alerts/{id}/resolve`
- правила реально создают алерты при нарушениях;
- история состояний алертов сохраняется.

## 9. Этап 6. Realtime Delivery

### Цель

Обеспечить live-обновления для интерфейса мониторинга.

### Задачи

- подключить `SignalR`;
- создать hub для устройств, событий и алертов;
- публиковать обновления после ingest;
- добавить подписки по группам устройств, локациям или клиентам;
- определить payload событий для фронтенда;
- предусмотреть защиту и авторизацию realtime-канала.

### Результат

- UI получает обновления в реальном времени;
- отпадает необходимость в постоянном polling для основных сценариев.

### Definition of Done

- изменения статуса и новые алерты приходят в realtime;
- клиент может подписаться на нужную область мониторинга;
- нет дублирования событий при стандартном сценарии работы.

## 10. Этап 7. Analytics

### Цель

Добавить аналитическую ценность поверх сырых данных.

### Задачи

- реализовать агрегаты `1m`, `1h` и при необходимости `1d`;
- считать `min`, `max`, `avg`, `count`;
- подготовить API для графиков и исторических выборок;
- внедрить фоновые jobs для агрегации;
- добавить базовое выявление аномалий;
- оптимизировать выборки для дашбордов.

### Результат

- система умеет показывать тренды и историю;
- появляются данные для аналитики и отчётности.

### Definition of Done

- графики строятся по агрегированным данным;
- запросы к истории не деградируют при росте объёма данных;
- аномалии фиксируются как события или алерты.

## 12. Этап 8A. Позднее внедрение Kafka

### Цель

Добавить событийную шину без переписывания основной бизнес-логики.

### Когда этот этап нужен

Этап стоит запускать не сразу, а при появлении хотя бы части признаков:

- объём телеметрии вырос настолько, что синхронный ingest становится узким местом;
- несколько подсистем должны независимо читать один поток событий;
- алерты, аналитика и интеграции нужно масштабировать независимо;
- нужна буферизация пиковых нагрузок;
- требуется повторное воспроизведение или replay-сценарии.

### Как встроить Kafka в текущую архитектуру

Рекомендуемый переход:

1. Сохранить внешний ingest API без изменений.
2. Перевести входящий поток телеметрии на публикацию в `Kafka`.
3. Поднять отдельный consumer для нормализации и сохранения телеметрии в `PostgreSQL`.
4. Использовать `outbox` для публикации внутренних событий в `Kafka`.
5. Вынести alerts, analytics и integrations в отдельных consumer-ов.
6. Оставить `PostgreSQL` источником истины для UI, API и realtime.

### Потоки, которые хорошо подходят для Kafka

- `telemetry.raw`
- `telemetry.validated`
- `device.state.changed`
- `monitoring.events`
- `alerts.created`
- `alerts.updated`
- `integrations.outgoing`

### Подготовительные условия

Чтобы внедрение прошло без болезненной переделки, до этого момента в проекте должны уже быть:

- application services вместо логики в контроллерах;
- отдельные обработчики ingest, state update, rules evaluation;
- `outbox` для интеграционных сообщений;
- идемпотентная обработка команд и событий;
- стабильные контракты входящих сообщений.

### Результат

- ingest отделяется от тяжёлой downstream-обработки;
- сервисы аналитики, алертов и интеграций могут масштабироваться независимо;
- система становится устойчивее к пиковым нагрузкам.

### Definition of Done

- ingest может публиковать сообщения в `Kafka`;
- consumer сохраняет телеметрию без потери целостности;
- алерты и интеграции получают события через `Kafka`;
- есть мониторинг consumer lag и ошибок доставки.

### Definition of Done

- CI проверяет сборку и тесты.

## 15. Приоритеты первой версии

Для `MVP` рекомендуется включить:

- `Foundation`
- `Data Layer`
- `Device Registry`
- `Ingest`
- `Device State`
- `Rules and Alerts`
- базовый `Realtime`

Можно отложить на вторую волну:

- расширенную аналитику;
- сложные интеграции;
- `Kafka`;
- продвинутые алгоритмы аномалий;
- глубокую multitenancy-модель;
- тяжёлую оптимизацию хранения.

## 16. Основные риски

- слишком рано усложнить архитектуру микросервисами;
- смешать ingest-логику, доменную логику и чтение для UI;
- недооценить объём телеметрии и рост таблиц;
- не заложить идемпотентность и retry-механику;
- не отделить сырые сообщения от нормализованных readings;
- строить realtime напрямую из тяжёлых запросов к БД.

## 17. Ближайшие практические шаги

Следующие действия для этого репозитория:

1. Убрать зависимость от шаблонного MVC-подхода и перейти к API-ориентированной структуре.
2. Создать доменные сущности и конфигурации EF Core.
3. Реализовать начальную схему `PostgreSQL` и миграции.
4. Поднять ingest endpoint и обновление `DeviceState`.
5. Добавить базовые алерты и realtime-обновления.

## 18. Итог

Этот roadmap задаёт практический путь от шаблонного ASP.NET Core-приложения к backend-сервису реального времени для мониторинга и анализа IoT-устройств.

При таком подходе команда сможет быстро получить рабочий `MVP`, а затем без радикальной перестройки развивать систему в сторону интеграций, аналитики и production-ready эксплуатации.
