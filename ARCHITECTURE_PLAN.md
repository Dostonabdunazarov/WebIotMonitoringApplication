# План архитектуры backend-части Web IoT Monitoring Application

## 1. Контекст

Текущий проект представляет собой базовый ASP.NET Core MVC-приложение с подключением `Entity Framework Core` и `PostgreSQL` через `Npgsql`.  
Целевая задача этого проекта: стать backend-частью системы мониторинга и анализа IoT-устройств в реальном времени.

Этот сервис должен быть частью общей системы и отвечать за:

- приём телеметрии от устройств или внешнего шлюза;
- сохранение и агрегацию данных;
- мониторинг состояния устройств в реальном времени;
- анализ событий, аномалий и пороговых значений;
- предоставление API для UI, внешних сервисов и административных инструментов;
- отправку уведомлений и генерацию системных событий.

## 2. Целевая роль сервиса в общей системе

Рекомендуется рассматривать этот проект как специализированный backend-сервис `Monitoring & Analytics`.

Внешние взаимодействия:

- IoT-устройства или gateway отправляют телеметрию;
- backend принимает сообщения, валидирует и сохраняет их;
- backend обновляет текущее состояние устройства;
- backend запускает анализ правил и аномалий;
- backend публикует события для UI и других сервисов;
- frontend или внешний BFF читает данные через HTTP API и realtime-канал.

## 3. Предлагаемая архитектура

Для этого проекта лучше использовать модульную слоистую архитектуру внутри одного сервиса.

Рекомендуемая структура:

- `Presentation`
  - REST API для устройств, операторов и интеграций;
  - realtime endpoint для live-мониторинга;
  - health checks и административные endpoints.
- `Application`
  - use cases;
  - orchestration бизнес-сценариев;
  - DTO, команды, запросы;
  - сервисы обработки телеметрии, анализа, уведомлений.
- `Domain`
  - сущности;
  - value objects;
  - доменные правила;
  - доменные события.
- `Infrastructure`
  - `Entity Framework Core`;
  - репозитории;
  - работа с `PostgreSQL`;
  - интеграции с очередью, кэшем, email, webhook и т. д.

Такой подход подходит для старта, потому что:

- не перегружает проект микросервисами на раннем этапе;
- позволяет отделить доменную логику от контроллеров и базы;
- упрощает дальнейшее выделение отдельных сервисов, если нагрузка вырастет.

## 4. Основные функциональные модули

### 4.1. Модуль реестра устройств

Отвечает за:

- регистрацию устройства;
- хранение метаданных;
- привязку к клиенту, объекту, локации, группе;
- хранение конфигурации, статуса активности и последней связи.

Основные сущности:

- `Device`
- `DeviceType`
- `DeviceProfile`
- `DeviceTag`
- `DeviceLocation`

### 4.2. Модуль приёма телеметрии

Отвечает за:

- приём входящих сообщений;
- валидацию формата и источника;
- нормализацию данных;
- сохранение сырых и обработанных показаний.

Источники могут быть такими:

- HTTP API;
- MQTT через gateway или отдельный consumer;
- message broker;
- batch-импорт из внешней системы.

### 4.3. Модуль текущего состояния устройства

Отвечает за быстрое получение текущей картины:

- online/offline;
- последнее значение датчиков;
- последнее время связи;
- состояние батареи;
- уровень сигнала;
- технические ошибки.

Этот модуль должен обновляться при каждом новом сообщении от устройства.

### 4.4. Модуль событий и алертов

Отвечает за:

- обнаружение выхода за пороги;
- фиксацию ошибок устройства;
- определение пропадания связи;
- генерацию алертов;
- подтверждение и закрытие инцидентов.

Основные сущности:

- `MonitoringRule`
- `Alert`
- `AlertStateHistory`
- `DeviceEvent`

### 4.5. Модуль аналитики

Отвечает за:

- агрегацию телеметрии по времени;
- расчёт min/max/avg;
- построение трендов;
- простое выявление аномалий;
- подготовку данных для дашбордов и отчётов.

### 4.6. Модуль интеграций

Отвечает за:

- публикацию событий во внешнюю систему;
- webhook-уведомления;
- email, Telegram или другие каналы оповещений;
- экспорт данных для других частей платформы.

## 5. Потоки обработки данных

### 5.1. Базовый realtime-сценарий

1. Устройство или gateway отправляет телеметрию.
2. Backend аутентифицирует источник.
3. Сообщение валидируется и нормализуется.
4. Сырые данные сохраняются в журнал телеметрии.
5. Обновляется текущее состояние устройства.
6. Проверяются правила мониторинга.
7. При необходимости создаётся событие или алерт.
8. Изменения публикуются в realtime-канал для UI.

### 5.2. Сценарий аналитики

1. Новые данные попадают в основную таблицу телеметрии.
2. Фоновый обработчик строит агрегаты по окнам времени.
3. Результаты сохраняются в таблицы агрегатов.
4. API отдаёт быстрые данные для графиков и отчётов.

- использовать `jsonb` для сырых payload и гибких метаданных;
- отделить сырые входящие сообщения от нормализованных показаний;
- хранить текущее состояние устройства в отдельной таблице для быстрого чтения.

## 7. Технологическая реализация в рамках текущего проекта

Так как проект уже на `ASP.NET Core` и `EF Core`, логично развивать его в этом стеке.

Рекомендуемый стек:

- `ASP.NET Core Web API` для backend API;
- `Entity Framework Core` для доступа к данным;
- `PostgreSQL` как основная БД;
- `SignalR` для обновлений в реальном времени;
- `BackgroundService` для фоновой обработки;
- `FluentValidation` для валидации входных запросов;
- `Serilog` для структурированного логирования;
- `OpenTelemetry` для трассировки и метрик;
- `Swagger/OpenAPI` для документации API;
- `Docker` для контейнеризации.

## 8. Предлагаемая структура проекта

## 9. API, которые нужны в первой версии

### Для устройств или gateway

- `POST /api/ingest/telemetry`
- `POST /api/ingest/batch`
- `POST /api/devices/{id}/heartbeat`

### Для операторского интерфейса

- `GET /api/devices`
- `GET /api/devices/{id}`
- `GET /api/devices/{id}/state`
- `GET /api/devices/{id}/telemetry`
- `GET /api/devices/{id}/alerts`
- `GET /api/alerts`
- `POST /api/alerts/{id}/acknowledge`
- `POST /api/alerts/{id}/resolve`

### Для realtime

- `SignalR Hub` или аналогичный websocket-канал:
  - обновление статуса устройства;
  - новые события;
  - новые алерты;
  - live-метрики по выбранным устройствам.

## 10. Нефункциональные требования

Нужно заложить с самого начала:

- отказоустойчивость при временных сбоях БД;
- идемпотентность при повторной отправке телеметрии;
- аудит критичных действий;
- логирование входящих ошибок и аномалий;
- масштабируемость чтения и записи;
- безопасность API и аутентификацию устройств;
- мониторинг самого сервиса.

## 11. Безопасность

Минимально рекомендуется:

- аутентификация устройств по API key, токену или сертификату;
- разделение доступа операторов и сервисных аккаунтов;
- ролевая модель для административных операций;
- шифрование чувствительных настроек;
- rate limiting для ingest endpoint;
- журналирование попыток невалидного доступа.

## 12. Поэтапный план реализации

### Этап 1. Подготовка основы проекта

- перевести приложение в формат backend-first;
- добавить слои `Domain`, `Application`, `Infrastructure`;
- настроить `PostgreSQL`, миграции и базовую конфигурацию;
- подключить логирование, `Swagger`, health checks.

Результат этапа:

готовый каркас backend-сервиса.

### Этап 2. Реестр устройств

- реализовать сущности устройств и справочников;
- сделать CRUD для устройств;
- добавить хранение статуса и последней активности.

Результат этапа:

система знает зарегистрированные устройства и их текущее состояние.

### Этап 3. Приём телеметрии

- реализовать ingest API;
- добавить валидацию и нормализацию payload;
- сохранять сырые сообщения и readings;
- обновлять `device_states`.

Результат этапа:

backend начинает принимать данные в реальном времени.

### Этап 4. Правила мониторинга и алерты

- реализовать правила порогов;

- создавать и обновлять алерты;
- добавить подтверждение и закрытие инцидентов.

Результат этапа:

появляется активный мониторинг и реакция на проблемы.

### Этап 5. Realtime для интерфейса

- подключить `SignalR`;
- отправлять обновления по устройствам, событиям и алертам;
- добавить подписки по группам устройств или локациям.

Результат этапа:

### Этап 6. Аналитика

- реализовать агрегаты по времени;
- подготовить API для графиков;
- добавить базовое выявление аномалий.

Результат этапа:

система не только показывает текущие значения, но и помогает анализировать поведение устройств.

### Этап 7. Интеграции и эксплуатация

- реализовать webhook или event publishing;
- добавить outbox-паттерн;
- настроить метрики, трассировку и алерты по самому сервису;
- подготовить контейнеризацию и CI/CD.

Результат этапа:

сервис становится полноценной частью общей платформы.

## 13. Практические рекомендации по реализации

- не смешивать контроллеры, `DbContext` и бизнес-логику в одном месте;
- не отдавать сырые таблицы напрямую в API;
- использовать отдельные команды для ingest, обработки правил и обновления состояния;
- выделить фоновые задачи для тяжёлых операций;
- закладывать `outbox` заранее, если нужны надёжные интеграции;
- хранить события мониторинга отдельно от телеметрии;
- сделать отдельный контракт для ingest, чтобы формат входящих данных был стабилен.

## 14. Как встроить Kafka позже

Для первой версии `Kafka` не является обязательной, но архитектуру лучше сразу строить так, чтобы её можно было добавить без переписывания доменной логики.

### Принцип встраивания

Нужно отделить:

- приём запроса;
- доменную обработку;
- сохранение данных;
- публикацию событий наружу.

Если это разделение соблюдено, то позже `Kafka` можно встроить как транспорт между этапами обработки, не меняя основную бизнес-логику.

### Целевой путь эволюции

Стартовая схема:

- `Device/Gateway -> HTTP Ingest API -> Application Service -> PostgreSQL -> SignalR/API`

Расширенная схема с `Kafka`:

- `Device/Gateway -> HTTP/MQTT Ingest -> Kafka topic -> Ingest Consumer -> PostgreSQL`
- `PostgreSQL + Outbox -> Outbox Publisher -> Kafka topic`
- `Kafka -> Alert Consumer / Analytics Consumer / Integration Consumer`
- `Consumers -> PostgreSQL / Notifications / External Systems / Realtime`

### Где Kafka лучше всего подключать

`Kafka` стоит добавлять не в UI-часть и не между API и обычным чтением данных, а в асинхронные потоки:

- входящий поток телеметрии;
- события мониторинга;
- генерация алертов;
- аналитические пайплайны;
- интеграции с другими сервисами.

### Рекомендуемые топики

Пример базового набора:

- `telemetry.raw`
- `telemetry.validated`
- `device.state.changed`
- `monitoring.events`
- `alerts.created`
- `alerts.updated`
- `integrations.outgoing`

### Как подготовить текущую архитектуру заранее

Чтобы потом подключить `Kafka` с минимальными изменениями, уже сейчас стоит:

- держать ingest-контракт отдельным от EF-моделей;
- оформлять обработку телеметрии через application services;
- не зашивать логику алертов внутрь контроллеров;
- хранить интеграционные события через `outbox`;
- делать отдельные обработчики для:
  - нормализации телеметрии;
  - обновления `DeviceState`;
  - проверки правил;
  - отправки уведомлений;
- проектировать идемпотентную обработку сообщений.

### Стратегия миграции без переписывания

Рекомендуемый путь перехода:

1. Оставить существующий HTTP ingest endpoint.
2. Вместо полной синхронной обработки начать публиковать входное сообщение в `Kafka`.
3. Вынести сохранение и обработку телеметрии в отдельный consumer.
4. Оставить `PostgreSQL` источником истины для UI и API.
5. Перевести alerts, analytics и integrations на отдельные consumer-процессы.

Таким образом:

- API остаётся стабильным;
- доменная логика переиспользуется;
- нагрузка распределяется по независимым обработчикам;
- появляется возможность масштабировать ingest и downstream-сервисы отдельно.

### Что важно предусмотреть при переходе

- идемпотентность consumer-обработки;
- message key по `device_id` или другому стабильному идентификатору;
- версионирование событий;
- retry и dead-letter strategy;
- наблюдаемость lag, retry и failed messages;
- чёткое разделение command flow и event flow.

### Когда переход оправдан

`Kafka` стоит добавлять, когда появляются признаки:

- ingest начинает упираться в синхронную обработку;
- несколько подсистем читают один и тот же поток событий;
- требуется буферизация пиков;
- аналитика и уведомления мешают основному потоку приёма данных;
- появляется необходимость replay или повторной обработки исторических событий.

Итоговый вывод:

сейчас систему лучше строить как модульный backend на `PostgreSQL`, `SignalR` и фоне обработки, но с `outbox`, идемпотентными handler-ами и выделенными application services. Тогда `Kafka` позже станет эволюционным шагом, а не архитектурной переделкой.

## 15. Что стоит сделать следующим шагом

Следующий практический шаг для этого репозитория:

1. Превратить проект из шаблонного MVC в backend API.
2. Создать доменные сущности `Device`, `TelemetryMessage`, `TelemetryReading`, `DeviceState`, `Alert`.
3. Расширить `ApplicationDbContext` и добавить первые миграции.
4. Реализовать ingest endpoint и базовое обновление состояния устройства.
5. После этого добавить realtime и правила алертов.

## 16. Итог

Для этого проекта оптимальная стартовая стратегия:

- один backend-сервис с чётким разделением слоёв;
- `PostgreSQL` как основная БД;
- realtime через `SignalR`;
- ingest + state + alerts + analytics как ключевые модули;
- постепенное развитие от простого мониторинга к аналитической платформе.

Такой подход даст быстрый старт разработки, не усложнит первую версию и при этом оставит хороший запас для масштабирования внутри общей системы.
